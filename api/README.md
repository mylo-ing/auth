<h1 align="center">myLocal API</h1>

<p align="center">
  Monolithic API for all myLocal micro services. Each myLocal service is kept in a separate subdomain and folder, so that they can be broken up into their own API when we have a need to scale.
</p>

---

## Project highlights

| Stack | Details |
|-------|---------|
| **Language** | Go 1.23
| **Web framework** | [Fiber v2](https://github.com/gofiber/fiber)
| **ORM** | [GORM](https://gorm.io/) + DigitalOcean Postgres 17
| **Cache / queue** | Redis 7 (cluster‑ready)
| **Container** | Multi‑stage `Dockerfile` (tiny Alpine runtime)
| **Kubernetes** | GitHub Actions → DigitalOcean K8s (Helm‑less manifests)
| **Docs** | OpenAPI 3 (Swagger UI)

## Local development

> **Prerequisites**
>
> * Docker and Docker Compose v2  
> * (Optionally) Go 1.23 if you want to run the binary outside Docker

```bash
# clone & enter repo
$ git clone git@github.com:mylo-ing/mylocal-api.git
$ cd mylocal-api

# start Postgres, Redis, and the API (hot‑reload via Air)
$ docker compose up --build
```

* **API available:** <http://localhost:3517>
* **PostgreSQL UI:** `localhost:5432` (user/pass in `docker‑compose.yaml`)
* **Redis:** `localhost:6379`

Every code change triggers live recompilation thanks to **cosmtrek/air** in the
`dev` image.

### Running unit tests

```bash
# inside the dev container
$ docker compose run --rm test
```

## Configuration

| Variable | Default (docker‑compose) | Purpose |
|----------|--------------------------|---------|
| `API_PORT` | `3517` | Container port Fiber listens on |
| `POSTGRES_DSN` | see `docker‑compose.yaml` | DB connection string |
| `REDIS_ADDR` | `mylocal_redis:6379` | Redis host:port |
| `SENDGRID_API_KEY` | _empty_ | e‑mail delivery in prod |
| `JWT_*` | demo secrets | Replace in prod secrets |

For local tweaks, copy `.env.example` → `.env` and adjust values; `docker compose` automatically picks them up.

## API documentation

* **Swagger UI** → <http://localhost:3517/docs> (local)  
* **JSON spec**  → <http://localhost:3517/docs/swagger.json>

The spec is generated by [`swag`](https://github.com/swaggo/swag) from Go doc
comments. To regenerate:

```bash
$ go install github.com/swaggo/swag/cmd/swag@latest
$ swag init -g cmd/server/main.go -o docs
```

CI regenerates the spec automatically on every push.

## Deployment pipeline (overview)

1. **Push → GitHub Actions** – lint, test, build image.  
2. **Image push** – `docker build‑push‑action` pushes to DigitalOcean Registry.  
3. **doctl apply** – manifests in `k8s/` are applied to the DO Kubernetes cluster.
4. **Vercel Frontend** consumes the API via `api.mylocal.ing`.

Secrets live in their respective platforms:

* GitHub Actions → deploy tokens (DB, Docker Hub, Digital Ocean, JWT...).
